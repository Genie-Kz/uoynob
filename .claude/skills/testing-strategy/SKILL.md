# テスト方針

テストを足すときは「価値」と「実行時間」のバランスを取る。むやみに本数を増やさない。

## ドメインロジックはユニットテスト

計算・検索・変換などの純粋ロジックは Vitest のユニットテストで網羅する（`src/domain/*.test.ts` など）。条件分岐や境界値を押さえる。これが土台で、最も費用対効果が高い。

## E2E は高価値な結合だけ

E2E（Playwright, `e2e/*.spec.ts`）は「**ユニットでは確認できない、結合でしか分からない部分**」に絞る。

- 対象になりやすい: ルーター遷移、データ取得、リアクティブな UI 配線、URL / localStorage の永続化、ブラウザ API、ガードやリダイレクト。
- 増やさない: スモークの薄い焼き直し、ドメイン計算の二重検証（ユニットで足りる）、データ依存で壊れやすい断定。
- ドメイン計算そのものは E2E で測らない（ユニットに任せる）。

## フレーキー対策

新規 E2E は `--repeat-each=2` などで複数回流し、安定して通ることを確認してから採用する。`page.url()` の断定はクエリの取り違えに注意（`new URL(...).hash` を正規表現で見るなど）。

## 実行コマンド

- ユニット: `npm run test`（CI と同じく `--project unit`）。
- E2E: `npm run test:e2e`（ビルド→preview に対して実行。`npm run check` には**含まれない**）。
