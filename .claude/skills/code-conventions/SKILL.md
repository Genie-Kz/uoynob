# コード・コメントの規約

`AGENTS.md` の規約を前提に、運用上の作法を補う。既存コードのデザイン・命名・コメント密度・イディオムに合わせて書く。

## コメント

- **日本語**で、意図・処理の流れ・分岐や例外処理の理由を多めに書く。冗長でも可。

## JSDoc を export 宣言に付ける

export された宣言（関数・型・定数・Vue の `defineProps`/`defineEmits` の各メンバー）には JSDoc コメントを付ける。

- IDE がホバー・補完で拾えるのは「**宣言の直上にあるブロックコメント** `/** ... */`」だけ。だから行コメント `//` ではなくブロックコメントにする。
- `import` を挟むとファイル先頭コメントは関数に紐付かないので、**export 関数の直上**に JSDoc を置く。
- 関数本体内のインラインコメントは `//` のままでよい（宣言ではないため）。

## 過度な抽象化を避ける

標準的で素直な解を優先する。凝った自前実装より、既存ツール・既存の仕組みに乗る方を選ぶ（例: ツールのバイナリを自前ダウンロードするより、コマンド直叩き＋既存の GitHub Action を使う）。安易なフラグ追加や、問題を覆い隠す辻褄合わせもしない（`AGENTS.md`）。指示どおりだと過度に複雑になる・UX を損なうなら代替案を出す。

## ビューとロジックの分離

ビューとビジネスロジックを密結合させない。計算・判定は純粋関数として書き、単一責任を意識する。ただし過度な共通化はしない。

## ファイルの置き場所

- `domain/`: 画面に依存しない計算・ルール・検索（純粋ロジック）。
- `features/`: 特定画面の状態管理・表示用ビューモデル・専用 UI。
- `presentation/`: ドメイン結果を UI コンポーネントが描画できる形へ詰め替える薄い変換層（計算でも描画でもない）。
- `shared/`: 共通 UI コンポーネント・アイコン・検索補助・データ取得層。
- `views/`: ルートに対応する薄いページ。
